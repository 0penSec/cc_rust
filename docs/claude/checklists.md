# 检查清单

> 提交前和代码审查时的必查项

---

## 提交前检查清单

在每次提交前，确保完成以下检查：

### 代码质量

- [ ] **格式化**: `cargo fmt --all` 执行完毕
- [ ] **Clippy**: `cargo clippy --all-targets --all-features -- -D warnings` 无警告
- [ ] **测试**: `cargo test --all` 全部通过
- [ ] **文档**: `cargo doc --all --no-deps` 无错误

### 文档

- [ ] 所有公共函数有文档注释
- [ ] 文档使用中文
- [ ] 复杂逻辑有行内注释
- [ ] 文档中的代码示例可运行

### Git

- [ ] 提交信息遵循规范（`type: 描述`）
- [ ] 单次提交只包含相关变更
- [ ] 没有遗漏的调试代码（`println!`, `dbg!` 等）

---

## 代码审查清单

审查自己或他人的代码时：

### 功能性

- [ ] 代码实现了预期功能
- [ ] 边界情况被正确处理
- [ ] 错误处理完善（使用 `Result` 而非 `panic!`）

### 可读性

- [ ] 命名清晰（变量、函数、类型）
- [ ] 函数长度适中（不超过 50 行）
- [ ] 注释解释"为什么"而非"做什么"

### 性能

- [ ] 没有不必要的克隆（`clone()`）
- [ ] 异步操作使用 `tokio::spawn` 而非阻塞
- [ ] 大数据集使用流式处理

### 安全性

- [ ] 没有硬编码的密钥或 token
- [ ] 用户输入被验证
- [ ] 文件路径处理安全（防止目录遍历）

### 测试

- [ ] 新增功能有对应测试
- [ ] 边界情况有测试
- [ ] 错误路径有测试

---

## 新功能开发清单

开发新功能时的完整流程：

### 设计阶段

- [ ] 明确功能需求
- [ ] 设计 API 接口
- [ ] 确定错误处理方式
- [ ] 考虑并发安全性

### 编码阶段

- [ ] 实现核心功能
- [ ] 添加文档注释
- [ ] 实现单元测试
- [ ] 运行检查清单

### 提交阶段

- [ ] 执行提交前检查
- [ ] 编写清晰的提交信息
- [ ] 推送到 GitHub
- [ ] 验证 CI 通过

### 发布阶段（可选）

- [ ] 更新版本号
- [ ] 更新 CHANGELOG
- [ ] 创建标签
- [ ] 推送到 crates.io（如果是库）

---

## CI 失败快速修复

| 失败类型 | 快速修复命令 |
|---------|-------------|
| 格式错误 | `cargo fmt --all && git add -A && git commit --amend --no-edit` |
| Clippy 警告 | 根据提示修复，然后 `git add -A && git commit --amend` |
| 测试失败 | 本地运行 `cargo test` 查看详情，修复后重新提交 |
| 文档错误 | 修复文档链接，重新提交 |

---

## 发布前检查清单

发布新版本时：

- [ ] 版本号已更新（`Cargo.toml`）
- [ ] CHANGELOG 已更新
- [ ] 所有测试通过
- [ ] 文档已更新
- [ ] 没有 TODO 或未完成的代码
- [ ] Git 标签已创建
- [ ] CI 全部通过
