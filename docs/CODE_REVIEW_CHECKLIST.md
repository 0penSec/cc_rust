# 代码审查检查清单

## 🔍 审查前准备

1. 拉取 PR 分支
2. 运行测试：`cargo test --all`
3. 检查 CI 状态

---

## ✅ 正确性检查

### 逻辑正确性
- [ ] 业务逻辑正确，无逻辑错误
- [ ] 边界条件处理完善（空输入、最大值等）
- [ ] 错误处理完整，所有 `Result` 都被处理
- [ ] 无竞态条件（并发代码使用正确的同步原语）
- [ ] 资源正确释放（实现 `Drop` trait）

### 类型安全
- [ ] 使用正确的类型（`String` vs `str`, `Vec` vs slice）
- [ ] 避免不必要的 `unsafe` 代码
- [ ] 泛型约束合理

---

## 🎨 代码风格

### 命名规范
- [ ] 函数/变量：snake_case
- [ ] 类型/结构体：CamelCase
- [ ] 常量：SCREAMING_SNAKE_CASE
- [ ] 模块名：snake_case

### 代码组织
- [ ] 函数长度合理（建议 < 50 行）
- [ ] 单个函数只做一件事
- [ ] 模块组织清晰，职责单一
- [ ] 导入分组：std → external → crate

### 注释和文档
- [ ] 公共 API 有文档注释（`///`）
- [ ] 复杂逻辑有行内注释（`//`）
- [ ] 示例代码可运行

---

## ⚡ 性能

- [ ] 无不必要的 `.clone()`
- [ ] 使用正确的集合类型：
  - 顺序访问 → `Vec`
  - 键值查找 → `HashMap`

- [ ] 异步代码使用正确的运行时（tokio）
- [ ] 锁使用合理，避免死锁
- [ ] 大数据集考虑流式处理

---

## 🔒 安全性

- [ ] 用户输入已验证
- [ ] 敏感数据（API key、token）未硬编码
- [ ] 敏感数据未记录到日志
- [ ] 文件路径处理安全（防止目录遍历）
- [ ] 无 SQL 注入风险（如使用数据库）

## 🧪 测试

- [ ] 新增功能有对应的单元测试
- [ ] 边界条件有测试用例
- [ ] 错误处理有测试用例
- [ ] 测试命名清晰（`test_功能_场景_预期结果`）

---

## 📋 常见代码问题

| 问题 | 坏代码 | 好代码 |
|------|--------|--------|
| 过度使用 unwrap | `file.read().unwrap()` | `file.read().await?` |
| 不必要的 clone | `data.clone()` | `&data` 或 `Arc::clone(&data)` |
| 阻塞操作在 async | `std::fs::read` | `tokio::fs::read` |
| 大类型传值 | `fn f(d: Vec<u8>)` | `fn f(d: &[u8])` |
| 忽略 Result | `file.write(buf);` | `file.write(buf).await?;` |
| 不必要的 mut | `let mut x = 5;` | `let x = 5;` |
| 冗长匹配 | `match opt { Some(v) => ..., None => None }` | `opt.map(\|v\| ...)` |

## 📝 审查评论规范

### 评论分类

**[BLOCKING]** - 必须修复才能合并
- 逻辑错误
- 安全问题
- 测试缺失

**[SUGGESTION]** - 建议采纳，但不强制
- 代码风格优化
- 性能改进建议
- 更好的命名

**[QUESTION]** - 需要作者澄清
- 不理解的设计决策
- 需要更多上下文

**[NIT]** - 小问题，可忽略
- 格式微调
- 拼写错误

### 示例评论

```
[BLOCKING] 这里的 unwrap 可能导致 panic，请使用 `?` 传播错误。
```

```
[SUGGESTION] 考虑使用 `HashMap::entry` 简化这段逻辑。
```

```
[QUESTION] 为什么选择 `RwLock` 而不是 `Mutex`？
```
